* Implementation Notes

** Design Decisions

*** Threading Model
- Used @MainActor for all JavaScriptCore-touching code
- CFMessagePort created on main thread with CFRunLoopGetMain()
- Direct JavaScript callback invocation (no async dispatch needed)

*** Bundle ID Extraction
- Dynamically extracted via Bundle.main.bundleIdentifier
- No hardcoding required

*** Readline Library
- Used libedit (BSD licensed) instead of GNU readline (GPL)
- Available via =#include <editline/readline.h>=
- Provides readline-compatible API

*** Execution Context
- Option B: Shared context with function parameter injection
- =_cli= and =print= injected as function parameters
- Global scope shared across instances

** v1.0 Scope Simplifications

Items deferred to v2.0:
- UserDefaults-based color configuration
- Persistent history across sessions
- Advanced tab completion (deep properties, methods)
- Legacy V1 protocol support
- Settings UI for CLI preferences

** Security Considerations

- CFMessagePort is local-only (no network exposure)
- Port names are global (any local process can connect)
- Arbitrary JavaScript execution allowed (intentional for CLI)
- Local user is the trust boundary

** Performance Characteristics

- CFMessagePort is synchronous (blocks until response/timeout)
- Message size limit ~1MB
- Main thread requirement for JavaScript execution
- Recursive call protection (max depth: 5)
